# ステップ 3.5: 実行アーキテクチャ設計

役割構成が確定したら、**どのように役割を動かすか**を設計する。
SubAgent と Agent Team を業務ごとに使い分けるハイブリッド構成を推奨する。

---

## 3.5-A: 実行パターンの解説と選択

まず2つの実行パターンの違いをテキストで簡潔に説明する:

```
📋 実行パターンの比較

SubAgent（委譲型）
  親 → 子に指示 → 子が結果を返す（一方向）
  コスト: 低い / 適用: 情報収集・定型処理・独立タスク

Agent Team（協働型）
  メンバー同士がメッセージを交換しながら進める（双方向）
  コスト: 高い（各メンバーに独立context）/ 適用: 議論・レビュー・設計
```

次に、定義済みの役割とワークフローをもとに**各業務に最適なパターンを自動提案**する。
提案のテーブル例:

```
| ワークフロー         | パターン    | 理由                                |
|---------------------|------------|-------------------------------------|
| タスクチェック        | SubAgent   | 秘書・SW・リサーチャーが独立に情報収集 |
| 設計レビュー         | Agent Team | エンジニア・QA・PMが相互フィードバック  |
| AI動向まとめ         | SubAgent   | リサーチャーが単独で完結               |
| 障害対応             | Agent Team | エンジニア・SW・PMがリアルタイム連携    |
```

## 3.5-B: オーケストレーター・ワーカー設計（SubAgent パターン用）

SubAgent パターンに割り当てたワークフローについて、以下を自動分析してテキストで提示する:

- **オーケストレーター候補**: 全体を統括し他の役割に指示を出す役割（例: PM、リーダー）
- **ワーカー候補**: 具体的なタスクを実行する役割（それ以外の役割）
- **並列実行可能なワーカー**: 互いに独立した調査・生成系タスクを担う役割
- **逐次実行が必要なワーカー**: 共有リソース（ブラウザ、ファイル、外部サービス）を操作する役割

---

## 3.5-C: チーム編成設計（Agent Team パターン用）

Agent Team パターンに割り当てたワークフローについて、以下を設計してテキストで提示する:

- **リーダー（lead）**: チームを統括する役割（Agent Team ではリーダーがチームメートをスポーンする）
- **チームメート（teammates）**: リーダーと協力して作業する役割
- **通信パターン**: どの場面で message（1対1）/ broadcast（全員）を使うか
- **コンフリクト防止策**: 同一ファイルを複数メンバーが編集する場合のルール

提案例:
```
🤝 Agent Team: 設計レビュー

  リーダー: PM（進行管理・最終判断）
  チームメート:
    - エンジニア (sonnet) — 実装観点のレビュー
    - QA (sonnet) — テスト観点のレビュー

  通信パターン:
    PM → broadcast: レビュー対象と観点を共有
    エンジニア ↔ QA: message で技術的な確認をやり取り
    各メンバー → PM: message でレビュー結果を報告
    PM → broadcast: 最終判断を全員に共有

  コンフリクト防止:
    - エンジニアは src/ のみ編集可
    - QA は tests/ のみ編集可
    - それ以外のファイル編集は PM に確認を取る
```

## 3.5-D: モデル割り当て（コスト最適化）

各役割のタスク特性と実行パターンをもとにモデルを提案する。
**Agent Team メンバーはコストが高い（独立context）ため、特にモデル選択が重要。**

判断基準:
- `opus` — 複雑な設計・判断・品質評価・最終意思決定を行う役割
- `sonnet` — 調査・コンテンツ生成・定型操作・レビューを行う役割
- `haiku` — ファイル検索・軽量な確認・フィルタリングのみを行う役割

コスト意識のルール:
- Agent Team のリーダーは判断力が必要なので `opus` または `sonnet`
- Agent Team のチームメートは `sonnet` を基本とし、軽量タスクなら `haiku`
- SubAgent のワーカーは `sonnet` を基本とし、単純作業なら `haiku`
- **全員を opus にしない**。opus は判断・設計が必要なロールに限定する

提案テーブル例:
```
| ロール       | SubAgent時 | Agent Team時 | 理由                    |
|-------------|-----------|-------------|-------------------------|
| PM          | opus      | opus (lead) | 最終判断・統括            |
| 秘書         | sonnet    | —           | 定型分類・API操作         |
| エンジニア    | opus      | sonnet      | Team時は議論参加のみ      |
| QA          | sonnet    | sonnet      | レビュー・検証            |
| リサーチャー  | sonnet    | —           | 調査・生成               |
| SW          | haiku     | —           | フィルタリングのみ         |
```
